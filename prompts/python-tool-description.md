Run Python in a browser-harness-compatible, raw-CDP browser environment.

This is the browser interaction tool. Use other tools for files, commands, plans, helper sessions, and completion; use this tool for browser navigation, browser inspection, clicks, typing, screenshots, downloads, uploads, network inspection, extraction, and browser-backed verification.

Browser work is shared mutable state, so keep browser actions sequential and deliberate. Use parallel local tools for independent read-only file or command inspection, but do not parallelize Python browser actions with other browser-changing tools.

CDP is the source of truth. Use raw CDP for basic browser control: `cdp("Page.navigate", url="https://...")`, `cdp("Runtime.evaluate", expression="document.title", returnByValue=True)`, `cdp("Input.dispatchMouseEvent", type="mousePressed", x=..., y=..., button="left", clickCount=1)`, `cdp("Input.dispatchMouseEvent", type="mouseReleased", x=..., y=..., button="left", clickCount=1)`, `cdp("Input.insertText", text="...")`, and `cdp("Page.captureScreenshot", format="png")`. Helpers are conveniences for host behavior, not a framework: `new_tab`, `goto_url`, `page_info`, `js`, `capture_screenshot`, `screenshot`, `screenshot_clip`, `emit_image`, `click_at_xy`, `fill_input`, `type_text`, `press_key`, `scroll`, `wait_for_load`, `wait_for_element`, `wait_for_network_idle`, `current_tab`, `list_tabs`, `switch_tab`, `ensure_real_tab`, `upload_file`, `drain_events`, `http_get`, `copy_artifact`, `set_final_answer`, `audit_artifact`, `get_final_answer`, `agent_workspace`, `load_agent_helpers`.

Do not import Playwright, Selenium, or Pyppeteer. Browser-harness workflow: first navigation should usually be `new_tab(url)`, not `goto_url(url)`, because `goto_url` mutates the active tab. Use screenshots as labeled temporal checkpoints: initial load, before/after meaningful clicks, scrolls, route changes, menus, dialogs, downloads, uploads, form submissions, and final verification. Prefer coordinate clicks for visible targets: `capture_screenshot` or `screenshot`, inspect the pixels, `click_at_xy(x, y)`, then screenshot again. Prefer capturing the action timeline inside one Python call when possible. Do not call `screenshot` repeatedly on an unchanged viewport; after a screenshot, act, inspect with CDP/JS, navigate, scroll, call `screenshot_clip(...)` for a different CSS-pixel region, wait for an async transition, or finish. Chrome hit-testing handles iframes, shadow DOM, and cross-origin content. Use `js(...)` and raw `cdp(...)` when coordinates are the wrong tool or helpers are incomplete. `js(...)` returns Python values, so use Python slicing/methods after the call; only use JavaScript methods inside the JavaScript expression.

To pass pixels directly to the next model turn, call raw `cdp("Page.captureScreenshot", format="png")`, `screenshot(label)`, `screenshot_clip(label, x, y, width, height)`, or `capture_screenshot(..., attach=True)`; use `emit_image(path)` for existing image files. Raw `Page.captureScreenshot` results are attached automatically. Multiple labeled screenshots are good when they form a temporal trace, not when they repeat the same unchanged page.

For final structured answers, call `set_final_answer(data, artifact_name=...)` instead of printing huge JSON. It persists the answer for `done`, emits a compact count/preview, and can write a user-visible artifact. Finish with `done(use_final_answer=true)` or `done(result="__use_final_answer__")`; do not pass a status sentence such as "final answer persisted", "need final done", or "need done use" as the final user-facing result. If the task asks for JSON/table output or provides a schema, pass that exact JSON/table object to `set_final_answer(...)`; do not pass a summary wrapper with only `artifact_path`, `record_count`, `schema`, or a sample row. For large extraction loops, checkpoint partial data to files/artifacts. Keep bulk extraction calls bounded: split by page/source/bucket, use per-request timeouts, write progress after each chunk, and print compact counts so a timeout can resume from saved state instead of restarting. Do not combine discovery, full crawl, detail fetching, audit, and finalization into one long tool call when it may approach the tool timeout. Save large benchmark outputs under `/home/user/outputs` (mapped to this task's isolated output directory) or `outputs_dir()`, then keep `done` compact with path, count, schema/columns, and one sample row only when the user did not request full inline JSON/table data. Use the browser to discover and verify; after stable endpoints, file URLs, XHR/fetch patterns, or pagination URLs are known, use `requests`, `http_get`, or threaded HTTP for bulk extraction.

Preserve source provenance for entity- or location-bound extraction. If the user asks for a specific store, address, venue, company, profile, product, filing, document, account, location, or source scope, record the selected source identity and selected source scope separately from search inputs, delivery destinations, filters, or user-provided claims. Include enough source evidence to prove the extracted data came from the right entity: current URL, page title, visible source/entity name, visible/source address or ID, and any distinct search/delivery destination fields. If a requested location/source says there are no local results and the page switches to "all locations", nearby, global, or broader fallback results, do not use those fallback records to present a complete scoped answer unless the user explicitly allowed fallback. Use `source_scope_evidence` in the audit with fields such as `requested_scope`, `requested_source_status`, `actual_scope`, `scope_match`, `fallback_used`, `fallback_allowed`, and `out_of_scope_record_count`. Do not overwrite a source identity or scope with the requested entity if the page actually selected a different source; either fix the source selection or explicitly report the mismatch.

Before finalizing large artifacts, use `audit_artifact(...)` to compute the checks the user implicitly cares about: duplicate count, missing required fields, per-bucket target counts, source evidence for claimed entities/locations, source-scope evidence for broadened/fallback sources, evidence for claimed top/best/highest/longest selections, and whether requested visual files are nonblank. Example: `audit = audit_artifact(records=rows, required_fields=['name','price'], dedupe_fields=['name','price']); set_final_answer(data, artifact_name='result.json', audit=audit)`. For nested outputs, pass flattened records such as `records=[item for category in menu['categories'] for item in category['items']]`. For entity/location-specific outputs, include source evidence: `audit_artifact(records=rows, required_fields=[...], dedupe_fields=[...], source_evidence={'requested_location': requested, 'source_url': page_info()['url'], 'source_page_title': page_info()['title'], 'selected_source_entity_name': visible_name, 'selected_source_entity_address': visible_address_or_id}, required_source_fields=['source_url','source_page_title','selected_source_entity_name','selected_source_entity_address'])`. If requested scoped sources were broadened, include source-scope evidence: `audit_artifact(records=rows, source_scope_evidence={'requested_scope': requested, 'requested_source_status': status_text, 'actual_scope': visible_scope, 'scope_match': 'match|partial|broadened|mismatch|uncertain', 'fallback_used': bool, 'fallback_allowed': bool, 'out_of_scope_record_count': n}, required_scope_fields=['requested_scope','actual_scope','scope_match'])`. If the answer has `selection_method`, `ranking_basis`, or similar top/best/highest/longest claims, audit the actual metric: `audit_artifact(records=selected, selection_metric_field='active_duration_days', selection_order='desc', selection_limit=5, selection_pool_records=candidate_pool, selection_key_fields=['library_id'])`. If per-record screenshots are deliverables, pass those paths as `visual_files` and set `unique_visual_files=True` unless identical images are genuinely expected. If exact engagement/rank metrics are unavailable, prefer an observed numeric proxy such as duration; do not claim page/result order proves performance unless the page visibly labels that sort. Do not replace missing required values with prose such as "not visible", "unavailable", "unknown", or "could not determine"; those are gaps, so either go back and extract the real value or report the gap explicitly in a final answer that says what remains incomplete. Do not pass hand-written audit dictionaries for audit-worthy artifacts; use the `audit_artifact(...)` return value. If `set_final_answer(...)` emits `audit=missing ready_for_done=False` or `audit_ready_for_done=False`, treat that as a failed pre-final check: fix the artifact/audit and call `set_final_answer(..., audit=audit)` again before using `done`.

Choose dedupe fields from the user's wording. If the user says "no duplicates" without saying "within each category", dedupe globally by the item/entity identity fields, not by category plus identity. Include category in the dedupe key only when duplicate entries are acceptable across categories or the user explicitly asks for per-category uniqueness.

If `done(use_final_answer=true)` reports that no persisted final answer exists, recover by calling `set_final_answer(...)` first. Do not finish with an empty placeholder when extracted data exists.

The Python namespace persists across tool calls in the same browser session, so reusable variables can be kept there. For reusable scripts, write small helpers to `.browser-use/agent-workspace/agent_helpers.py`; they are auto-loaded on later calls, and can also be loaded with `load_agent_helpers()`. For final or expensive extracted data, do not rely on a remembered variable name after compaction; call `set_final_answer(...)` and/or write an artifact.
